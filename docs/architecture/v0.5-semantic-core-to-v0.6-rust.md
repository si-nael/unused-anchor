# v0.5 Semantic-Core Closure Before the v0.6 Rust Substrate

Status: active architecture boundary; Gate A released as bounded `v0.5.6`

Date: 2026-07-17

## Purpose

The remaining `v0.5.x` line must leave one portable, connected execution meaning before `v0.6.x` begins a Rust implementation. This is not a demand to complete every future Bubble feature in TypeScript. It is the smaller but strict requirement that the concepts already treated as the central world kernel no longer depend on accidental Node, JSON, or TypeScript behavior.

Rust is the first native physical backend. It is not the ontology, source language, universal machine model, or final substrate of Bubble. TypeScript remains the reference oracle and authoring/inspection implementation until exact differential conformance justifies transferring execution authority.

## Why binary layout is not the first layer

A byte stream can be compact and fast to decode while preserving the wrong meaning. The stable order is:

1. define exact semantic values and relations
2. define a typed machine-independent semantic graph
3. define canonical ordering and identity rules
4. encode that graph in a versioned binary envelope
5. interpret or compile it on one or more physical backends

The binary envelope must preserve unknown sections without executing them, distinguish semantic order from irrelevant storage order, and avoid freezing research concepts that remain open. File bytes, in-memory layout, and world identity are three different relations.

## Remaining 5.x gates

### Gate A: finish the lifecycle seam through the language boundary

Bounded `v0.5.6` has typed birth, retirement, lineage, residue, inactive-world rules, persistence, inspection, and replay. A separate strict `causal bubble` profile lowers explicit exact source declarations into that same contract. The older descriptive spawn/collapse IR remains unchanged because its effects have different meanings.

Closure requires:

- one typed lowering path into the same `spawn-world` and `retire-self` meaning used by the causal runtime
- no second lifecycle interpretation hidden in materialization tooling
- rejection of lifecycle rank regression and unanchored foreign-world reads at the authored-language boundary
- canonical run, inspection, record, replay, and tamper fixtures produced through the lowering path

Implementation state: completed and released as bounded `v0.5.6`. The strict lowering, ordinary causal CLI path, focused negative tests, canonical run/inspection/record/replay fixtures, complete repository boundary, and evidence-density audit pass. Gate B is the next 5.x semantic seam.

### Gate B: connect worlds only through a typed membrane

The `v0.5.6` lifecycle correctly rejects cross-world field reads because the current kernel has no honest transfer relation. A multi-world Bubble cannot remain a catalog of isolated local executors, so the next core seam is typed cross-world anchor and sea transport.

Closure requires:

- explicit source and target anchor endpoints, world identities, ports or capabilities, and transferred content
- causal provenance from source event through membrane crossing to target effect
- positive- and negative-sea interaction or residue accounting without treating either sea as a moral scalar
- inactive-world, cut-anchor, identity-failure, direction, and replay rules
- no direct foreign field read/write, host callback, or World-Will teleportation

This gate need not solve unrestricted networking, continuous fields, or every global topology. It must establish one exact nonlocal causal relation from which wider transport can grow.

### Gate C: specify the portable semantic graph

The stable core must have an implementation-independent representation covering:

- exact integers, rationals, symbols, identifiers, records, sets, and intensional references used by the selected slice
- local laws, guards, field effects, lifecycle effects, anchor/sea transfer, causal frontiers, branches, commits, provenance, and honest unresolved results
- canonical semantic ordering where order matters and canonical set treatment where it does not
- versioned extension sections that round-trip without silent deletion or execution
- an explicit distinction among semantic graph identity, canonical storage identity, and backend program identity

This graph is provisionally called BIR, but its name and byte tags do not carry semantics by themselves.

### Gate D: establish conformance and cost evidence

Before Rust begins, TypeScript must emit a compact implementation-independent fixture corpus. Each fixture contains input semantic graph, expected event ledger, selected continuation set, final state, lifecycle and transfer provenance, commits, digest, and expected resource-boundary verdict.

The cost model records external realization work rather than Bubble time. It should at least distinguish:

- decoded semantic nodes
- exact arithmetic operations or operand growth
- guard and law evaluations
- frontier and branch candidates
- materialized intensional coordinates
- state nodes allocated versus structurally shared
- bytes read, retained, and hashed

Optimization may change these costs but not semantic results. An exhausted executor returns the same typed `undetermined` boundary rather than a convenient partial world.

## What may wait until 6.x or later

The following do not block the first Rust kernel unless a Gate A-D counterexample requires them:

- deliberative agency, protagonist identity, relationships, or narrative projection
- dynamic self-authored schema and population generation
- general coinduction, continuous physics, or every infinite domain
- Phase 2 external observers and comparative benchmarks
- recursive self-construction and verified successor generation
- typed oracle, transfinite execution, or physical hypercomputation

Their idea obligations remain open. Deferring them from the transition gate is not rejection or philosophical closure.

## 6.x entry condition

Rust work may begin only after Gates A-D have explicit artifacts, adversarial tests, replay evidence, research reconciliation, and a complete repository verification. No single passing example or package-version edit satisfies the condition.

The first Rust slice should decode one stable BIR fixture, validate it, execute one bounded vertical causal path, and reproduce the TypeScript event ledger and digest exactly. It should not begin by porting every TypeScript file or by adding self-modifying code.

## Preserved boundaries

- No universal clock is introduced by host profiling.
- No host scheduler selects a world branch.
- No optimization removes a lawful possibility without a typed in-world quotient or collapse law.
- No binary constant, random source, anchor, sea, or World Will hides an oracle.
- No Rust object layout becomes the definition of same-world identity.
- No 5.x closure claim completes OB-001 or authorizes Phase 2.
